## 1. Vision
A Grade 8 Mathematics **study & practice hub** for a teacher with one or more students. Students sign in by name, open a topic, and climb a guided, self-paced lesson. The teacher sees who practiced what, where they struggled (by concept), exam-readiness, and can assign homework. Warm, professional, age-appropriate. Low stress, high rigor, MCAP-ready. Multi-student.

## 2. Non-negotiable principles
1. **Consult the textbooks; use the best, no filler.** All source PDFs live in the project folder and **must be consulted** every unit. Scan the folder, read the relevant chapters/items, and select the strongest explanations and exercises *from them*. Only substitute a created item when it is genuinely better.
2. **Curriculum ∪ exam (union, never trade-off).** Cover every concept the grade's curriculum requires, AND weave in real MCAP items on top as capstones. Never drop curriculum to chase the test.
3. **Flow builds up.** No advanced concept before its prerequisite. Representation before operations. Within a lesson: Learn → Guided → Practice → Apply → Exam.
4. **Hints sparingly — and strategy-only.** Only on multi-step/hard items. Never on trivial questions or exam capstones. A hint must coach the next move (name the strategy, point at the setup); it must **never finish the computation or contain the final answer** — a worked solution behind a Hint button is an answer key, and it silently invalidates mastery tracking because hint use is not logged.
5. **One structure, no duplicates.** Reuse the single engine and shared data contract. Never build a parallel system.
6. **Don't bog the student down.** Strategic and lean; quality over quantity.
7. **Professional, not "AI".** No emoji, no gradient/clichéd styling. Clean, real-product look.
8. **On grade.** Match Grade 8; shelve/rebuild anything above or below grade.

## 4. Shared data contract (localStorage — same browser/origin)
**Every hub owns its own prefix.** Grade 8 is `STORE_PREFIX='g8.'` (Grade 7 stays `g7.`); each module carries a `G7_STORE` that **must match its hub exactly** — a module whose prefix disagrees writes progress the dashboard never reads. Both hubs declared `g7.` until 19 Jul 2026, so on any device that had opened both (the tutor's, always) the two grades shared one roster/data/pins/current/teacherPass — the cause of a deleted record reappearing, a Grade 7 student on the Grade 8 dashboard, and roster removals doing nothing. `migrateStore()` carries this hub's own students across once per device (`g8.migv`) — **copy, never move** (Grade 7 still runs on `g7.`), roster-filtered, never overwriting a key the new namespace already holds; `seedv`/`passv`/`urlv` are deliberately not copied so seeds re-apply cleanly. Keys below are named by suffix (`<prefix>roster` etc.): `roster`, `current`, `subject` (last-selected subject), `seedv` (JSON of the last-applied `ROSTER_SEED`, so seed changes merge exactly once per device), `device` (device binding: the last student to sign in successfully; the sign-in screen then shows only their tile, with a "Someone else? Show all students" link for shared devices — binding follows the most recent sign-in and survives reloads), `passv` (last-applied `TEACHER_PASS_SEED` — the deployment passcode OVERWRITES any locally set teacher passcode once per seed value, reclaiming devices where a student set one during the old forced setup; store the seed as `atob('<base64>')` so it isn't readable in view-source), `urlv` (last-applied `SHEET_URL_SEED`), `pints` (per-pin last-write timestamps for sync LWW), `pins` (`{ "<name>": "<4-digit PIN>" }` — per-student light gate, plaintext, set by the student on first sign-in, teacher-resettable; not real security), `sheetURL`, `teacherPass`, and `data` with the per-student/per-topic tree (tree of steps, totalSteps, sectionTotals, lastPracticed, attempts, correct, skillStats, struggles, exam, responses). Teacher homework is per-subject under `students[name].assignments[subjectId]` (legacy `students[name].assignment` is still read and treated as Mathematics). A module writes the topic tree; the hub reads it.

**Retention & analytics fields (all additive — absent on legacy records, and every reader falls back).** The topic record also carries: `reviewStreak`/`reviewDay` (`MR-1` phase-2 — the consecutive-session rung and the calendar-day guard; when absent the hub infers a proxy streak from accuracy/evidence/shakiness); `acqFirst`/`acqCorrect`/`retFirst`/`retCorrect` (`AN-4` — first attempts bucketed by whether the topic was **due when the session started**, i.e. acquisition vs retention); and `levelStats{1..4}` (`AS-4` — attempts/misses per curriculum level, derived from each card's own phase tag: 1 Foundational, 2 Target, 3 Exam, 4 Stretch). `skillStats[skill]` is `{attempts, misses, last, streak, day}` — `last`/`streak`/`day` (`MR-1` phase-3) are what let a **single skill** come due inside an otherwise-solid topic. All of these ride the existing topic record, so they inherit the v1.5 `lastPracticed`-LWW sync with **no protocol change**. Never rename the `g7.` prefix or an existing key shape; back-compat is explicit. These fields originate in the Grade 7 engine and are inherited here.

**Module review entry:** a module accepts `?review=<skill>` (`MR-1` phase-3) and opens directly into a retrieval set — up to 4 of that skill's already-authored Target/Exam cards, cleared for a fresh attempt, with the rest of the lesson hidden. It **moves** cards rather than removing them (step totals stay honest) and **never writes the tree**, so the mastery bar stays monotonic. An unknown or unauthored skill falls back to the normal lesson. This exists because `restoreProgress()` deliberately re-fills completed work with its own answers — fine for "review your work", an answer key for a spaced revisit.

**Multi-subject model:** the hub config is `SUBJECTS → units → topics`. A topic's `id` **is** the storage key its module writes (`G7_TOPIC_ID`). Existing Math topics keep their bare ids (`number-system`, `expressions-equations`) for back-compat; **every NEW subject's topics must be prefixed with the subject** (e.g. `sci.life`) so two subjects can never collide in a student's data. Because hub + modules share one localStorage, they must be served from the **same origin** (a single deployment) for one combined dashboard.

**Per-student subject visibility:** `STUDENT_SUBJECTS` (edit-per-deployment constant next to `ROSTER_SEED`, e.g. `{Ayodeji:['math']}`) limits which subjects a student sees; a student not listed sees all. Seeded, **not synced** — like `ROSTER_SEED`, so every device is identical. It gates exactly three student-facing entry points (`renderSubjectTabs`, `firstSubject`, `curSubject`) and **nothing else**: `subjectById`/`eachTopic`/`topicMeta` stay unfiltered because they are data plumbing (sync merge, homework topic ownership), and the **teacher dashboard is deliberately unfiltered** — the teacher sees every subject for every student, and hidden-subject data is retained, never deleted. The dashboard instead **labels** non-enrolment (a `hidden from <name>` chip on the subject heading, and an empty state that says *why* rather than the misleading "No activity yet"). **Do not make the dashboard filter by enrolment:** it is the audit surface every claim in a report is checked against, and a default-hidden view there is the same silent-loss failure mode as an unrendered homework item. `curSubject()` must re-check permission because `<prefix>subject` is a **device-level** key: on a shared device it can hold another student's choice, and honouring it would render the wrong subject's content under a correct-looking tab bar. A single visible subject hides the tab bar. **This is visibility, not security** — module files remain reachable by direct URL.

**To add a unit:** set `G7_TOPIC_ID` (subject-prefixed for new subjects), `G7_TOPIC_TITLE`, the `G7_SKILLS` map; author content; add one entry to the right subject's `units` in the hub `SUBJECTS` array (status `available` + `file`).

## 5. Module baseline (anatomy)
Header (title · unit · standard · objectives) → hub bar → progress → **5–8 sections**, each = objective + teach card with **one worked example** + a climb of step-locked questions → **module check** (mixed, exam-style) → **make-your-own** challenge → export/reset.

**Phase tags (only chip vocabulary):** Learn / Guided / Practice / Apply / Exam / Stretch.

**Six item formats (all in the template):** fill-in (accepts equivalent fractions), single multiple-choice, multi-select, two-part (A unlocks B), constructed-response, and **click-to-plot** (`data-plot="line"` number line / `data-plot="grid"` coordinate grid). Mark exam-grade items with `data-exam="1"`.

**The plot is the 6th format, and it is currently used by nothing** (ported 21 Jul 2026 in `478816d`; corrected here the same day — §5 said "Five" and §3 pointed at the wrong path). It is a pointer/keyboard UI painted over a hidden `.ans-input`, so the ordinary engine still does the grading, locking, restore and review-reset — which is exactly why it must not be hand-rolled per module (§7.2). The invariant most worth guarding is that a click on a *locked* step writes nothing: **the plot is a second input surface over the step-lock ladder.** `tests/plot_format.test.js` (ported 21 Jul 2026, 26 assertions, mutation-checked on five paths) holds it to that, and its Part B sweeps **any module that adopts the format** so the first unit to use it cannot diverge from the template.

**Choose the format from the standard's verb.** A standard that says *construct*, *graph*, *sketch* or *represent on a number line* is not met by a multiple-choice item that asks the student to pick the right picture — that passes `exam_coverage` while assessing recognition in place of production, which is bar-lowering (§2.4's sibling failure). Standards in this grade whose verb demands the plot: `8.AT.A.1a` (graph proportional relationships), `8.NOS.A.2` (locate on a number line), `8.AT.B.4` (solution set on a number line), `8.AT.D.11` (sketch a graph from a description), `8.DS.B.2` (construct a scatter plot). See `MCCRS_2025_DUAL_CODING.md`.

**qid convention (required):** every qcard's `data-qid` is `<section>-<n>` (e.g. `3-2` = section 3, item 2). The number before the first `-` drives `sectionTotals` and the teacher dashboard's per-section bars — never use any other scheme, and never duplicate a qid.

**No access gate (removed 2026-07-17).** The old shared `g8gate`/`g8.gate` password overlay is gone from the hub, every module, and the template — a student lands straight on the student sign-in, which is the only gate (per-student name + PIN, below). `g8.gate` is dead; do not reintroduce a page password.

### The Number System — skill tags used (drives the struggle dashboard)
`rational-decimal` (rational numbers & decimal expansions), `repeat-fraction` (repeating decimal → fraction), `irrational` (identify irrational numbers), `roots` (square & cube roots), `estimate` (approximate irrationals between integers / truncating), `compare` (compare & order on a number line), `expr-est` (estimate value of expressions), `reasoning` (reasoning & error analysis).

### Functions — skill tags used (drives the struggle dashboard)
`func-def` (what a function is — one output per input), `func-rep` (representing & comparing functions), `linearity` (linear vs non-linear), `graph-read` (slope & y-intercept from a graph — **MCCRS 2025 `8.AT.D.9`, new**), `func-build` (building a linear function from a rate and an initial value), `qualitative` (describing & sketching graphs), `reasoning` (reasoning & error analysis).

## 6. Homework model
Auto-suggested from each student's weakest concepts; optional teacher-set per-student assignment; the make-your-own challenge as the creative anchor. Verification is automatic via timestamped tracking + dashboard.

### Homework plans (`<student>.plans.json`)
Sets are authored **tutor-side as a JSON file** and published via **Teacher → Settings → Publish plans**. Plans are **never committed** — `.gitignore` carries `*.plans.json` and `homework-plans/`, because they name students and this repo is public. Published plans land at `data.students[<name>].hwplan[<subjectId>]`; per-set completion at `…hwstate[<subjectId>]`.

```json
[{ "student": "<roster name>", "subject": "math",
   "sets": [{ "id": "wk-2026-07-20", "label": "Set 1 — …", "releaseOn": "2026-07-20",
              "items": [ { "ref": "module", "topic": "number-system", "qid": "4-1" },
                         { "ref": "task", "text": "…returned to the tutor as a photo." } ] }] }]
```

**Three item kinds** (`ref`) — anything else is rejected:
- **`module`** — needs `topic` + `qid`; the topic must be one this hub owns. The only kind the engine marks.
- **`bank`** — needs `id`. Reserved for the question bank; **renders nothing until the bank ships.**
- **`task`** — needs `text`. An **out-of-band** item: written/research work the student returns to the tutor directly (a photo). Rendered as a `.hw-task` instruction line that names its own return route; **deliberately never engine-marked.**

**Accounting rules — do not "fix" these, they are load-bearing:**
- `hwSetProgress()` counts **`module` items only**. So a `task` can never block a set reaching "done ✓", and a **task-only set correctly never auto-completes** (0/0 is not done).
- `hwLinks()` collapses **consecutive** same-topic `module` items into ONE `?q=a,b,c` link — one trip, not three. **Item order is preserved on publish and interleaving a non-module item splits the link**, so put `task` items at the start or end unless a split is intended.
- `hwDoneMap()` is deliberately **not** read from the module's topic tree: that records any past practice, so a set would show progress for work done weeks earlier and could auto-complete a homework the student never did. Modules write homework markers only when opened through a `&hw=` link.
- Homework links call `g7revReset()` on each card, so **re-assigning an already-completed qid is a genuine closed-book attempt, not an answer key** (same invariant as `?review=`). Re-assigning old items for spacing is safe and intended.
- `releaseOn` (`YYYY-MM-DD`, optional) locks a set until that date; once released it **stays open forever** — no catch-up lockout. ISO dates compare as plain strings; no date parsing.
- Task `text` is `esc()`d. Plans are tutor-authored but still untrusted input.

**Publishing fails closed:** one bad item publishes *nothing*, because a half-published week is worse than an unpublished one. A `no-cors` write can never report failure, so **always confirm by reading the plan back as teacher** — "looks published but isn't" is a failure mode this system has already been bitten by. **If the engine changed, push before publishing** (§11): an old live hub rejects item kinds it does not know.

## 7. Build & verify discipline
1. **Consult the sources first** (scan folder, read chapters/items); build from the best.
2. Author from the template; lean, best-of. Never hand-roll the answer-checking handlers — they carry the integrity invariants (§8).
3. **Verify every build:** `node --check` every extracted `<script>`; check structure (`</html>` present, balanced braces/parens, qid uniqueness, every qid covered in `G7_SKILLS`); recompute every answer key by hand or by code.
4. **Behavioral verification (required for hub or engine changes):** run the standing jsdom suite `tests/behavioral_test_suite.js` (76 assertions: gate, seeded roster, sign-in → PIN → app, device binding, teacher modal + gated Settings, Escape, passcode takeover, homework decay/done, CR review, module wrong → retry → correct with cooldown, shuffle + noshuffle, copy-block, exam first-attempt-only, fraction equivalence, locked steps, persistence, section report card, hint integrity). Usage: `npm install jsdom` once, then `node tests/behavioral_test_suite.js <folder with the HTML files>`. Extend the suite whenever engine behavior changes. A page that parses can still be dead on arrival (v1.3 found a null-binding crash that blanked the whole hub) — only a behavioral test catches that.
5. **Large HTML files (>~35KB): do NOT use editor Write/Edit tools** — an after-save reformat can truncate the file mid-`<script>`. Regenerate via a bash `python3` heredoc doing string `.replace()` on the intact file, then re-verify (step 3–4). Assert each replacement matched exactly once.
6. Keep this standard updated; log every build in §10.
7. **Sourcing fidelity — when you lift a released or textbook item** (codified 21 Jul 2026 from `TODO.md` P2; every rule was learned expensively, mostly in Grade 7):
   1. **`pdftotext` is an index, never the evidence.** It drops figures, collapses drop-downs, merges table rows and silently omits text. **Render the page and read it** before lifting an item *or un-lifting one* — a negative search is not proof of absence. Rendering is `python3 -c "import fitz; …get_pixmap(dpi=140).save(…)"` (PyMuPDF is installed; `pdftoppm` is not). This has now bitten four times, most recently on the MCCRS crosswalk, where extraction merged two standards and would have mis-assigned a deletion.
   2. **Reproduce the stem verbatim** and rebuild the figure to its stated dimensions. If a figure cannot be read off with certainty, the item is **blocked, not approximated**.
   3. **Recompute every key independently** (`fractions.Fraction`, never floats) *before* checking it against any published key. Two independent derivations must agree.
   4. **An adapted item loses the label.** Change the numbers or the response format and it is no longer that released item — retitle it `Exam-style ·` and record it. Copying the *format* of a real item, label included, is the exact pressure that produced 19 falsely-labelled capstones in Grade 7; see `MCAP_PROVENANCE.md`.
   5. **Record the lift.** Released items go in `MCAP_PROVENANCE.md` with their citation, checked by `tests/mcap_provenance.test.js`. **Textbook** lifts carry no exam label — a wrong attribution there cannot mislead a student about what they are sitting — but they are still recorded, so a future editor can tell what is the book's and what is ours.

## 8. Engine versioning (anti-dilution)
Engine: **Engine v1.5** (v1.4.3 + the shared cloud-sync layer, 2026-07-04). Additional invariants:
- **Cloud sync is offline-first and optional:** with no sync URL everything works per-device exactly as before. Backend = ONE Google Apps Script web app + Sheet (`Starter_Kit/HUB_Sync_Apps_Script.gs`, setup in `Starter_Kit/HUB_Google_Sheet_Setup.md`) **shared by ALL hubs** (SAT, Grade 7, Grade 8, …): every push/pull carries a hub namespace (`SYNC_HUB_ID='grade8'` here; each module carries `G7_SYNC_HUB`), so programs never collide and one student can be in several. The URL is baked in via `SHEET_URL_SEED` (applied once per value via `g7.urlv`) — zero-setup on every device.
- **Sync semantics:** modules push the student's own topic ~1.5 s after work (debounced) and pull-merge on load; the hub pulls on boot + every 60 s. Merge rules: topics = per-topic newest-`lastPracticed` wins; PINs = last-write-wins vs per-pin local timestamps (`g7.pints`); assignments = LWW by `ts`/`mts`/`doneTs`; teacher "Mark reviewed" travels as a per-topic review map overlaid onto responses by response `ts`. Hub pushes: PIN create/reset, assignment save, Mark done, Mark reviewed, and any local-newer topics found during a pull. Roster and passcodes do NOT sync — they deploy via seeds so every device is identical.
- Reads use JSONP (Apps Script has no CORS); writes use `no-cors` POST. All sync failures are silent no-ops.

Engine v1.4.3 (v1.4 + the Grade 7 back-port, 2026-07-04 — the two grade hubs are at engine parity; improve in one place, re-propagate deliberately). Additional invariants:
- **Settings is teacher-gated:** the top-bar gear routes through the teacher modal (`enterTeacher('settings')`) before `openSettings()` — students can never reach the passcode/sheet/backup controls. Escape closes any open modal.
- **Copy-block (modules):** question-card text is unselectable and copy/cut are suppressed (`user-select:none` on `.qcard`; inputs/textareas exempt).
- **MC/MS shuffle (modules):** option order shuffles per load for 3+-option groups; `data-noshuffle` opts out. **Grade 8 rule:** exam capstones keep their real MCAP order, and numeric/semantic scales (ascending money, One/None/Infinitely-many) keep their order — all marked `data-noshuffle`.
- **Hints are strategy-only** (§2.4): no hints on exam items, no worked answers behind Hint buttons.
- **Section report card:** every module injects "Your section scores" (per-section bar → done/total · %, row links to the section) under the overall progress box via `renderSecReport()`; the hub's student topic cards show the same per-section mini bars as the teacher dashboard. Section sort/labels tolerate non-numeric qid prefixes.

Engine v1.4 (v1.3.1 + the polish/anti-cheat layer, 2026-07-01). Additional invariants:
- **Answer obfuscation:** `data-answer` values are `k1:<base64>`; correct flags are per-option `data-k="k1:<base64 of n|0/1>"` (no `data-correct` anywhere). The engine decodes via `ansVal()`/`optCorrect()` (with plain-attribute fallback). **When authoring, write plain `data-answer`/`data-correct`, then run the encode pass before shipping** — never ship plaintext answers. This is a casual deterrent (view-source), not security.
- **MC guess-cooldown:** after a wrong MC click, remaining options lock for ~1.6s (discourages brute-force clicking).
- **Teacher access is a proper modal** (create-passcode mode on first use, enter mode after) — never `prompt()`/`alert()`.
- **"Skills to review" and homework suggestions decay:** a skill counts only with ≥2 misses AND ≥25% miss ratio, so mastered skills clear as clean attempts accumulate.
- **Homework loop closes:** student "Mark done" toggle (`assignments[sid].doneTs`); teacher sees "marked done" in the dashboard; saving a new assignment resets it.
- **CR review loop closes:** teacher "Mark reviewed" per written response (`responses[i].reviewed`); student sees "✓ Reviewed by your teacher" in the module.
- Storage caps: struggles ≤120, responses ≤60 per topic. Number inputs blur on scroll-wheel. No "Clear All" claim in the answer guide.

Engine v1.3 (v1.2 + the assessment-integrity layer, 2026-07-01). The bullets below are **invariants — any new or edited module/hub must preserve every one**:
- **Hub — zero-setup, locked roster + typed-name sign-in (2026-07-17):** the app is client-side, so every device starts empty. `ROSTER_SEED` (edit-per-deployment constant next to `SUBJECTS`) is **authoritative**: on every boot the roster is set to exactly the seed and any stray name (and its PIN) is pruned, so only seeded students can ever sign in. Adding a student = edit `ROSTER_SEED` and redeploy (ad-hoc Manage-roster adds don't persist). A student **types their name** — resolved to the canonical spelling, case- and spacing-insensitive — and the roster is **never displayed**, so no student sees who else is enrolled; a used device binds (`g7.device`) and thereafter shows only "Continue as <name>", no list, no "show all".
